# ADR 017 — 数据治理管道 + 数据源可插拔 + 商用授权闸门

- **状态**: 已接受（BoyangJiao 签收 2026-06-18：「就先这么干」）
- **日期**: 2026-06-18
- **作者**: Claude (Opus 4.8) + BoyangJiao
- **相关 ADR**: 006（`@arc/ui` / data-sources adapter 抽象 — 源可插拔的基础）、007（真实链路不可绕过）、010（dev 缓存信任）、011（多源 fallback + AKShare）、013（UI wrapper 所有权与 vendor swap — 同构思路用于数据源）
- **相关 spec**: `benchmark-comparison-stage-3.md`（指数对标 #9）、`insights-enrichment-stage-3.md`
- **触发**: 2026-06-18 数据源选型调研发现三件事 ——（1）**Tushare Pro 服务协议为「个人、非商用」**（`doc_id=405`），AKShare 是无授权的公网爬虫封装、聚宽同为研究授权 → 三者**均不可用于商用分发**；（2）美股历史走 Alpha Vantage 免费档（~25 次/天）频繁失败，已临时切 AKShare `stock_us_daily`（dev 兜底）；（3）客户端因 RLS 写不进 `price_snapshots` → 共享缓存填不上 → 每次打实时 → 限额更快耗尽。深挖必盈 / 沧海后确认：**两家都缺 CN 开放式公募基金「历史净值」**（必盈基金仅实时、沧海"基金"= 场内 ETF）。

---

## 背景

Arc 是**面向 C 端、商用分发**金融数据的产品，数据腿为：A股 · 港股 · 美股+ETF · **CN 开放式公募基金净值** · 指数 · FX · crypto。当前 dev 期数据源谱系（ADR 011 + 本对话）：

| 腿                    | 现状（dev）                                     | 商用牌照         |
| :-------------------- | :---------------------------------------------- | :--------------- |
| CN / 港 / 基金 / 指数 | Tushare（付费档）/ AKShare 兜底                 | ❌ 非商用 / 灰区 |
| 美股历史              | AKShare `stock_us_daily`（临时，替下故障的 AV） | ❌ 灰区          |
| 美股实时              | Finnhub                                         | ✅               |
| FX                    | Frankfurter                                     | ✅ 免费          |
| crypto                | CoinGecko                                       | ✅ 免费          |

调研结论把问题钉死成**两个必须分开处理的层面**：

1. **数据质量（工程）**：AKShare 本质是实时爬虫封装 —— 接口不稳、字段不统一、脏数据、无历史快照。这是可以用工程手段解决的。
2. **分发牌照（法律）**：AKShare/Tushare/聚宽 的底层数据是未授权抓取/非商用授权。**清洗、归一、入库、改字段名只改变格式，不改变来源合法性** —— 在商用 app 里分发给用户仍是未授权再分发，叠加中国《反不正当竞争法》「搭便车」风险 + 站点 ToS 违约。"很多量化团队这么干" 不适用 Arc：他们多为**内部/自用、不对外分发**（暴露极低），而 Arc 是**对外分发**（暴露最高）。

> ⚠️ 关键认知：**治理管道解决 #1，绝不解决 #2。** 把 AKShare/Tushare 焊死成生产源、以为"清洗过就安全"，等于把合规债务包进了管道。

---

## 决策

### D1 — 建服务端「数据治理管道」(Data Governance Pipeline)

采集 → 清洗/归一 → **落自有快照库** → app **只读**。复用并扩展现有表（`price_snapshots` / `fx_rates` / `portfolio_value_snapshots`）。这本质上就是 Q3 讨论的「服务端代理 + 共享缓存 + 自有 DB」：

- 行情对所有用户是同一份 → 按「标的 × 日期」缓存一次，调用量 ≈ **不同标的数 × 刷新频率**，**与用户数无关**（成本/限流与规模解耦）。
- 历史快照由管道沉淀，根治 AKShare「无历史」痛点。

### D2 — 源与管道**解耦**，源**可插拔**（ADR 006 / 013 同构）

管道是 Arc 的资产、长期不变；底层 source adapter 可热插拔：

| 阶段                      | 管道     | 数据源（喂管道）                                          |
| :------------------------ | :------- | :-------------------------------------------------------- |
| **dev / dogfood（现在）** | 同一条   | AKShare / Tushare（**个人非商用，合法**）                 |
| **上架 commercial**       | **不变** | **换授权源**：沧海 / 东财 Choice / 同花顺 iFinD / EODHD … |

app 端永远只读自有库，切源时 app 零改动。

### D3 — 清洗 ≠ 洗白牌照（硬约束）

治理管道**只解决数据质量**。**AKShare/Tushare/聚宽 禁止出现在 production 数据源配置中**；不得以"已清洗/已入库"为由在商用版分发其数据。

### D4 — 密钥下沉服务端 + 服务端写快照库

- 数据源 token **不再以 `EXPO_PUBLIC_*` 进客户端 bundle**（现状是泄露面）；移到服务端管道。
- 快照库由**服务端**写入；客户端**只读**（根治当前 RLS 客户端写入被拒 → 缓存填不上的问题）。

### D5 — CN 公募基金历史净值 = 专项采购缺口

必盈（仅实时）/ 沧海（仅 ETF）/ EODHD/Polygon（无 CN 开放式）均不覆盖。上架前需**单独**向 **东财 Choice / 同花顺 iFinD**（性价比优于 Wind）授权采购该腿，或就该数据取得商用授权。

---

## 后果

**利**：dev 立即稳定（自有库兜底 AKShare 抖动）+ 历史快照沉淀 + 客户端无密钥/无直连 + 成本与用户数解耦 + 上架换源 app 零改动 + 合规风险可控（源在管道底层一处切换）。

**弊**：需建并运维一套服务端管道（采集/调度/清洗/落库）—— 工作量与运维成本；上架前**必须**完成「源切授权 + 公募基金专项采购」，否则带合规债上线。

**风险分级**：私测/极小范围风险低，但随用户数/营收/曝光/应用商店审核/融资尽调线性放大 —— 属"长大卡脖子"型债务。故以下发版闸门为**硬门**。

---

## 发版闸门（上架前必办 — Release Gate）

- [ ] production 配置中**移除全部 AKShare / Tushare / 聚宽 源**（含 client 内任何 `EXPO_PUBLIC_*` 数据源 token）
- [ ] 每条数据腿（CN股 / 港 / 美 / **公募基金净值** / 指数 / FX / crypto）均指向**有商用授权**的源
- [ ] **CN 公募基金历史净值** 授权采购落实（Choice / iFinD 或等价）
- [ ] 数据源 token 全部服务端化；客户端 bundle 不含任何数据源密钥
- [ ] 快照库由服务端写入、客户端只读（RLS 策略相应调整）
- [ ] 数据来源标注 / 「仅供参考，可能延迟」/ 免责声明 符合各授权源的署名要求

> 闸门未全绿，不得对外公开分发（App Store / 公测 / 收费）。
